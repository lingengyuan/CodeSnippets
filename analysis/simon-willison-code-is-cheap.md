## 30秒 TL;DR

> AI Agent 把"打字写代码"的边际成本压至接近零，这**摧毁的不是代码本身的价值，而是我们过去几十年建立在"时间稀缺"上的工程决策直觉**。好代码（有测试、有文档、正确处理边界）的成本没有同步下降——下降的只是"打字"成本。所以新时代的挑战是：把过去因时间成本而放弃的所有事情重新估价，同时在质量保证上投入同等或更多的人类判断力。

---

## 概念总览

| 概念 | 核心思想 | 适用场景 |
|------|---------|---------|
| **代码打字成本 vs 好代码成本** | 两者不再同步——前者趋零，后者未变 | 任何关于"值不值得实现"的决策 |
| **宏观工程直觉的失效** | 精细规划/功能ROI评估都建立在时间稀缺假设上 | 项目规划、功能优先级 |
| **微观决策的重置** | "重构值不值？测试值不值？"这类问题的答案全变了 | 日常 code review、技术债决策 |
| **并行 Agent 的乘数效应** | 一人同时跑多个 Agent = 实现+重构+测试+文档同步进行 | 复杂功能开发 |
| **异步试错习惯** | 直觉说"不值"时先 prompt 一下，最坏损失几分钟 | 任何"感觉太小"的改进想法 |

## 深读

### 核心论点的结构

这篇文章的论点是一个**三层逻辑**，但 Simon 只明确写出了前两层，第三层藏在字里行间：

```
层 1（事实）：编码成本大幅下降
层 2（推论）：旧的工程直觉建立在"编码成本高"的假设上，假设失效 → 直觉失效
层 3（隐藏）：你现在的"这不值得做"直觉是系统性的、跨组织的偏差，而不只是个人偏见
```

第三层的含义：如果你和你的团队都在用同样的旧直觉做判断，那么**所有人都系统性地少做了某些事情**。第一个重建直觉的人/团队，会发现一批被旧成本模型错误定价为"不值得做"的事情，现在变得极其值得。

### 好代码的定义（作为清单的价值）

Simon 列出的 8 个"好代码"维度值得仔细对待，因为这不只是定义，而是一张**Agent 质量审计清单**：

| 维度 | 可被 Agent 自动完成？ | 仍需人类判断？ |
|------|---------------------|---------------|
| 代码能运行 | ✅ Agent 可执行验证 | ❌ 边界 case 定义 |
| 我们**知道**它能运行（有测试） | ✅ Agent 可写测试 | ⚠️ 测试充分性判断 |
| 解决了正确问题 | ❌ 需求理解是人的事 | ✅ |
| 优雅处理错误/边界 | ⚠️ Agent 能实现但需指定 | ✅ 边界定义 |
| 简单可维护 | ⚠️ Agent 倾向过度工程 | ✅ |
| 有回归测试 | ✅ | ⚠️ 测试覆盖率判断 |
| 文档与代码同步 | ✅ | ⚠️ 文档准确性判断 |
| 为未来变更留余地（不违背 YAGNI） | ❌ Agent 不懂产品路线图 | ✅ |
| 各种 -ility（安全、可观测……） | ⚠️ 部分可自动 | ✅ 选哪些 -ility 是人的判断 |

**关键洞察**：这张表显示，Agent 降低了"执行"成本，但所有需要**定义问题边界**的事情仍然是人的工作。代码便宜了，但**"问正确的问题"的能力**价值翻倍。

### "异步试错习惯"的真正含义

Simon 写的那句话值得单独放大：

> "任何时候你的直觉说'别做那个，不值这时间'，就异步跑一个 agent——最坏情况是十分钟后回来发现真的不值那些 token。"

这不只是一个操作建议，它是一个**认识论建议**：在新成本结构下，用实验代替直觉判断。旧直觉是从旧成本环境里归纳出来的，它是有系统偏差的，而实验的成本足够低，不需要通过直觉做预筛选了。

## 心智模型

> **"把你的直觉当成缓存，而不是真理"**：旧的"值不值得做"直觉是在旧成本结构下训练出来的缓存。Agent 让底层成本发生了结构性变化，缓存需要失效重建。在新习惯稳定之前，凡是直觉说"不值"的，都值得做一次低成本实验来验证。

**适用条件**：
- 你在主动使用编码 Agent（而非只是问答式 LLM）
- 任务的"打字成本"确实是过去的主要瓶颈
- 质量要求在"生产级代码"范围内（而非草稿或 PoC 随便写写）

**失效条件**：
- 当瓶颈不是打字，而是**理解问题**（例如：复杂领域建模）
- 当团队协调成本、代码审查成本、部署成本是主要摩擦（这些没有因 Agent 下降）
- 当"好代码"的标准本身不清晰——Agent 会生成代码，但如果你不知道好代码长什么样，Agent 也帮不了你

## 非显见洞见

### 洞见 1：成本下降不对称——"试错"比"执行"受益更多

- **洞见**：传统软件工程中，"试错"（做一个 spike 看看能不能走通）和"执行"（正式实现）的成本比大约是 1:3 到 1:5。Agent 让两者都下降了，但**试错受益更大**——因为 spike 本来就不需要好代码，只需要能运行，而 Agent 最擅长的正是"快速跑通 happy path"。
  - 所以：用 Agent 做 spike 的 ROI 比用 Agent 做生产代码的 ROI **高得多**
  - 所以：最优策略是：Agent 跑 spike → 人判断 spike 是否值得执行 → 人+Agent 做有质量保证的执行
  - 因此可以：**在任何不确定的技术方向上，把"先用 Agent 做 10 分钟 spike"变成默认动作**，而不是先讨论 30 分钟再决定要不要做

### 洞见 2：好代码定义中"我们知道它能运行"比"它能运行"更昂贵

- **洞见**：Simon 把"代码能运行"和"我们知道它能运行"列为两个**不同的维度**。这个区分在 Agent 时代变得比以往任何时候都重要。
  - 所以：Agent 非常擅长生成"看起来能运行"的代码，甚至实际上也能运行，但**在你没有检查之前，你不知道**
  - 所以："知道它能运行"这件事——即测试、验证、证明——的成本**相对**上升了（因为代码产出速度变快，但验证速度没有同步变快）
  - 因此可以：**投资在 agent 可自动执行的验证机制**（自动化测试、CI）比以往任何时候都更值——它是唯一能跟上 Agent 代码产出速度的质量保证手段

  **蕴含链**：
  ```
  Agent 产出速度↑ → 代码产出量↑ → 未经验证代码↑ → 质量风险↑
  → 所以：自动化测试的 ROI = (验证成本节省) × (代码量增幅)
  → 所以：项目的第一个任务应该是建立 CI，而不是写业务代码
  → 因此可以：在 Agent 项目启动时，把"先建 CI pipeline"作为标准 SOP
  ```

### 洞见 3：并行 Agent 打破了"编码时间"作为资源约束的假设

- **洞见**：传统项目计划里，"工程师时间"是最稀缺的资源。并行 Agent 让同一个工程师可以同时推进多条线（实现/重构/测试/文档），这**使得"工程师数量"不再是吞吐量的主要瓶颈**。
  - 所以：团队规模与功能产出量之间的正比关系被打破了
  - 所以：新的瓶颈变成了：**工程师能维持多少上下文（同时督导多少个 Agent 任务）**
  - 因此可以：用"工程师可同时维持的有效 Agent 并发数"作为新的工程能力衡量指标，而不是"代码行数/人天"

  **隐含假设**：这个洞见成立，前提是 Agent 任务之间耦合度不高。高度相互依赖的任务无法真正并行，因为 Agent 需要等待彼此的输出。

### 洞见 4：YAGNI 在 Agent 时代需要重新校准

- **洞见**：Simon 在"好代码"定义里同时强调"设计要为未来变更留余地"和"遵守 YAGNI（不过度设计）"。这两个原则在旧成本结构下形成了一个平衡点，但在 Agent 时代这个平衡点移动了。
  - 所以：过去"不值得实现这个功能的 v2"是因为 v2 实现成本高；现在 v2 实现成本接近零
  - 所以：YAGNI 原则的核心诉求（不要因为预测未来而增加当下复杂度）在"当下复杂度可以随时用 Agent 重构"的情况下，**力度应该减弱**
  - 因此可以：在 Agent 项目里更偏向"先实现最简单的版本，知道它可以被随时重构"，而非"设计一个扛住未来变化的架构"

## 反模式与陷阱

- **陷阱 1："代码便宜了，所以我不需要想清楚需求"**
  代码便宜了，但改错误方向的代码的**机会成本**没有降低（用 Agent 时间做了错误的事，不是零成本）。
  → 正确做法：用 Agent 做 spike 来**澄清需求**，而非在需求模糊时直接让 Agent 实现

- **陷阱 2："Agent 写了测试，所以测试一定够"**
  Agent 会写测试，但会优先写最容易写的测试（happy path），而非最有价值的测试（边界 case、错误路径）。
  → 正确做法：明确要求 Agent 测试边界条件和错误路径；用 coverage 工具作为客观指标

- **陷阱 3："所有决策都应该交给 Agent 试"**
  "异步试错"的建议适用于代码层面的不确定性，不适用于**产品方向**的不确定性。花 10 分钟让 Agent 实现一个功能，不能替代对这个功能是否解决了用户问题的判断。
  → 正确做法：把 Agent 试错用于技术可行性判断，用用户研究做产品方向判断

- **陷阱 4："代码便宜了，所以 code review 可以减少"**
  代码产出速度↑ × code review 频率↓ = 未审查代码量指数级增加。Agent 生成的代码仍然可能包含安全漏洞、逻辑错误、性能问题。
  → 正确做法：code review 密度应该与代码产出量成比例，甚至更高

- **陷阱 5：把"旧直觉"与"新直觉"混用而不自知**
  最危险的状态是：对某些事情用了新习惯（让 Agent 试），但对另一些事情仍在用旧直觉（这不值得花时间），却无法意识到哪个是旧直觉在作怪。
  → 正确做法：建立一个显式的"我选择不做这件事，原因是..."日志，定期回顾哪些"不值得做"其实只是旧直觉的惯性

## 隐含假设分析

Simon 的文章基于若干未明说的假设，这些假设的边界决定了文章的适用范围：

| 假设 | 假设成立时 | 假设不成立时 |
|------|-----------|-------------|
| Agent 生成的代码质量足以作为起点 | ✅ 大多数 CRUD、工具类、胶水代码 | ❌ 高度专业化领域（密码学、实时系统、底层 OS 代码）|
| 工程师有能力审查 Agent 输出 | ✅ 工程师熟悉该领域 | ❌ Agent 在工程师不熟悉的领域写代码 |
| "打字时间"是主要成本 | ✅ 标准 Web/App 开发 | ❌ 需求分析、架构设计、团队对齐是主要成本的项目 |
| Agent 工具的经济成本合理 | ✅ 当前 API 定价 | ❌ 高频调用、超大 context 项目可能 token 成本不可忽视 |

## 衍生项目想法

### 想法 1：工程决策日志（Decision Log with Old-vs-New Habit Audit）

**来源组合**：[本文"旧直觉是系统偏差"洞见] + [已有 `python/snippet_manager.py` 的 LLM 调用能力]

**为什么有意思**：
最难建立新习惯的地方在于：旧直觉是**无意识**的，你不知道自己什么时候在用旧直觉做决策。如果有一个轻量级工具，让你在说"不做这个功能"的时候记录原因，LLM 可以分析这个原因是"合理的成本决策"还是"旧成本模型下的直觉偏见"，就能帮你识别哪些"不值"其实只是惯性。具体地说：输入"我决定不实现 X，因为 Y"，LLM 返回"这个理由在新成本结构下是否仍然成立，还是需要用 Agent 试一下"。

**最小 spike**：
1. 写一个 CLI 脚本（`python/decision_audit.py`），接受 "决定不做 X，原因 Y" 的输入
2. 用一个简单 prompt："你是一个帮助工程师识别过时决策直觉的助手。以下决定是否建立在'代码编写时间稀缺'这个旧假设上？如果是，估计用 Agent 快速验证这个方向需要多少 token / 时间。"
3. 把有价值的决策记录存到 `~/.decision_log.jsonl`，方便定期回顾
4. 预计完成时间：2小时

---

### 想法 2：Spike 专用 WORKFLOW 模板（Fast-Fail Mode）

**来源组合**：[本文"10分钟异步 spike"模式] + [已有 `templates/WORKFLOW.md.template` 的编排框架]

**为什么有意思**：
当前 WORKFLOW 模板是面向有明确需求的任务执行。本文提出的"先 spike 再决定做不做"需要一个不同的 workflow：只跑 happy path、显式跳过测试和文档、设置 5-10 分钟 timeout、输出格式是"技术可行性评估 + 建议下一步"而非可交付代码。这个 spike 模板与生产级模板分离，可以防止 spike 过度成熟（花了 1 小时写了个"spike"，实际上写了可以生产的代码）。

**最小 spike**（讽刺地，这本身就是一个 spike）：
1. 在 `templates/` 下创建 `WORKFLOW_SPIKE.md.template`
2. 关键差异：`quality_mode: spike`（跳过测试/文档要求）、`timeout_minutes: 10`、`output_format: feasibility_report`
3. 在 `mini_symphony.py` 中增加对 `quality_mode: spike` 的支持：任务完成后只输出"✅ 可行 / ❌ 不可行 / ⚠️ 部分可行"摘要
4. 预计完成时间：3小时

### 想法 3：CI-First Agent 项目启动器

**来源组合**：[本文"验证成本相对上升"洞见（洞见 2 蕴含链）] + [已有 `python/sandbox_execute.py` 的隔离执行能力]

**为什么有意思**：
Agent 时代，代码产出速度↑，但验证速度不变，导致"未验证代码积压"成为新的技术债。解决这个问题的关键是：**在项目第一行业务代码之前就建立 CI**。一个"CI-First 项目启动器"可以：给定一个项目描述，自动生成 GitHub Actions 配置、pytest 框架、预提交钩子，确保从第一个 commit 开始就有自动化验证。这把"好代码的维度 2（知道它能运行）"从"做完后补测试"变成了"从一开始就内置"。

**最小 spike**：
1. 写一个 `python/ci_first_init.py`，接受"项目类型"（Python lib / Python CLI / HTML tool）
2. 生成并执行：初始化 git repo、创建 `.github/workflows/test.yml`、创建 `tests/__init__.py` + 一个空的测试文件、配置 pre-commit
3. 验证：push 到 GitHub 后 CI 自动触发并通过（即使只有一个空测试）
4. 预计完成时间：半天
